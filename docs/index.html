core features

What it does

Six major capabilities, each designed around how the shop floor actually works.

🔍

QR-driven routing

Scan once to start, scan again to end. The system detects whether it's a start or end scan automatically — no mode-switching, no buttons to remember.

📊

Live WIP board

Every active job is slotted into the column for its current process. Amber = waiting for operator pickup. Refreshes on every scan submission.

📈

Stats & drill-down

Sidebar shows Completed / In-Progress / Idle per station. Filter by Today, This Week, Month, or Custom Range. Double-click to see job-level detail.

📄

Travel Sheet PDF

Landscape A5 PDF with QR code, process sign-off grid, job info fields, and printed-by metadata. Built with ReportLab, layout mirrors the UI exactly.

🔐

Planning module

Password-protected supervisor page for creating new job routings — WSR, drawing, qty, ordered process sequence. Jobs appear in Pending immediately.

🕒

History & correction

Double-click any history row to load it into the input panel. Supports corrections to Mc No, PIC, remarks, and timestamps via Submit.

scan flow

How a job moves through the floor

From planning to completion — the full lifecycle driven by QR scans.

1

Supervisor creates job routing

Planning page (password-protected) — WSR, drawing, qty, ordered process sequence written as rows to DATA.xlsx on the shared network drive.

2

Job appears in Pending

get_wip_status_map() finds no Time Start anywhere for this WSR → job sits in the Pending column of the WIP board.

3

Operator scans QR at station

get_active_row() finds the lowest-sequence row without Time Start → routes to the correct station page automatically.

SCAN #1 — START
  • Info panel filled (WSR, Drawing, Process)
  • Operator types Machine No & PIC
  • Submit → writes Time Start to Excel
  • Job moves to station column (amber)
SCAN #2 — END
  • Auto-reads Mc No & PIC from DB row
  • Writes Time End immediately
  • get_next_process() → route to next
  • WIP board refreshes, stats update
4

Automatic routing to next station

Next station page opens automatically with the job pre-loaded. Repeat until all sequences complete.

5

Job complete — drops from WIP board

All sequences have Time Start + Time End. Job exits the active board. History tables retain full records. PDF travel sheet printable anytime.

config-driven design

One object, one station

All 8 machining stations share identical UI logic. A single ProcessConfig entry generates a fully-themed, fully-wired page. Adding a new station is a 10-line change.

ProcessConfig(
  name = "GRINDING",
  page_index = 10,
  route_key = "grinding", # matched against Process column in Excel

  col_mc_no = "Mc No Grinding",
  col_pic = "PIC Grinding",
  col_time_start = "Time Start Grinding",
  col_time_end = "Time End Grinding",
  col_remarks = "Remarks Grinding",

  tag_prefix = "GRINDING", # widget tag namespace
  accent = (128, 0, 77, 1), # RGBA theme colour per station
  accent_light = (231, 141, 190, 0.28),
)

// to add a new process station: add one ProcessConfig to the registry list. zero other file changes.

engineering learnings

What building this taught me

Real problems that don't come up in tutorials, and how I dealt with them.

Shared-file concurrency

openpyxl's load/save cycle is not atomic. Works fine in this single-writer environment, but multi-writer would need a lock file or migration to SQLite. A deliberate tradeoff for zero-infrastructure deployment on a facility network drive.

Qt layout in resizable windows

PySide6 geometry management for dynamically-created widgets (QTableWidget inside a custom layout) requires careful cleanup and rebuild on resize. A debounced resizeEvent pattern at 150ms prevents layout thrashing during window drag.

Excel as a self-healing schema

Columns are created on first write if missing (_ensure_col), making the schema self-healing. But column order isn't guaranteed across installs. A migration step would be cleaner for a larger team — acceptable tradeoff here for operator simplicity.

Config-driven UI factories eliminate duplication

The single ProcessConfig → create_process_page_ui() pattern cut roughly 1,000 lines of duplicated code across what would have been 8 near-identical files. Adding a new station went from copy-paste-edit to a 10-line change. The pattern paid for itself on the first extension request.

Sequence-aware row targeting

When a WSR has the same process at multiple sequences (e.g. two grinding passes), get_active_row() finds the lowest-sequence row without a Time Start. Without this, a naive match would write to all rows for that process simultaneously, corrupting the data.
